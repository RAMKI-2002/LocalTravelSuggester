1. Overview

Local Trip Suggester is a FastAPI service that recommends tourist places for a city, tailored to the live weather, the user's preference (a free-text intent like "peaceful places with views"), and the user's locality (used for distance + travel-cost estimates).

The interesting parts (and the parts to highlight in an interview):

  • LLM intent extraction - free-text prompts like "want to eat something" or "feeling tired" are translated into a structured {category, search_keywords, mood} via a rule-based fast path with a Bedrock fallback for ambiguous phrasings. Without this every prompt would fetch the same generic "tourist attractions" pool from Foursquare.
  • Async fan-out - geocoding + intent extraction run in parallel; weather + places run in parallel right after. asyncio.gather all the way down.
  • Layered fallback & runtime circuit-breaker - Foursquare → OpenStreetMap Overpass → stale cache; weather → stale cache; Bedrock → rule-based template. The Foursquare client carries a class-level circuit-breaker: one "no credits" 429 trips it and every subsequent request in the process skips Foursquare entirely (Overpass becomes the primary).
  • Overpass mirror failover - 3 mirrors (overpass-api.de, overpass.kumi.systems, overpass.private.coffee) tried in order on transport errors / 5xx / 429. The main Overpass instance is heavily overloaded; failing over silently means the user almost never sees an empty result.
  • Anchor-first queries - we always geocode the city (and locality if given) and query Foursquare / Overpass with (lat,lng) + 25 km radius. We then drop anything >30 km from the anchor. This kills the "Pune query gave Lonavala results" class of bug that comes from inconsistent OSM admin boundaries.
  • Hybrid LLM role - the rule-based ranker does the broad cut (cheap, deterministic, explainable); the LLM curates the final picks from the top-2x shortlist in one Bedrock call and writes a one-line reason for each. Names are validated against the shortlist so the LLM cannot hallucinate a place.
  • Diversity rule - the ranker caps how many places of the same primary category go into a single result set, so different intents (peaceful vs adventure vs history) actually surface different places.
  • Live observability - JSON structured logs, per-request UUID, a ring-buffer log endpoint, and /health/detailed that probes every dependency in parallel (with three states: ok / disabled / down, so a kill-switched provider shows amber, not red).

5. Code flow (function-level)

A request to POST /suggest-trip flows through these functions in this order. File paths are clickable in the repo:

  1. app/main.py: create_app() - mounts routes + static UI; lifespan logs startup
  2. app/utils/logger.py: RequestIdMiddleware - stamps every request with a 12-char UUID, sets contextvar, returns it as x-request-id
  3. app/api/routes_trip.py: suggest_trip_endpoint() - Pydantic validates the body, then delegates to the service
  4. app/services/trip_service.py: suggest_trip() (the orchestrator)
    1. Stage 1: resolve. Geocodes city + (optionally) locality AND extracts intent via intent_parser.extract_intent() - all three in parallel. Anchor = locality > city > none. Intent is a TripIntent dataclass with category, search_keywords, mood, source ("rule" / "llm" / "default").
    2. Stage 2: fan-out. Provider selection lives here. If PlacesClient.is_disabled() (config kill-switch OR runtime breaker open), Overpass becomes the primary - no wasted call to a known-dead provider. Otherwise launches WeatherClient.get and PlacesClient.search_tourist_places(lat, lng, radius=25km, query=intent.query_string, cache_namespace=intent.category) in parallel.
    3. Stage 3: fallback chain. If FSQ raises UpstreamError with a "no credits" 429, the breaker trips (PlacesClient._billing_disabled = True) and we call OverpassClient.search_tourist_places(lat, lng, radius=25km) which itself fails over across 3 mirrors.
    4. Stage 4: distance filter - drops anything >30km from the anchor. Hard safety net for inconsistent OSM admin boundaries.
    5. Stage 5: ranker.rank_places() scores each candidate (0.20*weather + 0.45*intent + 0.15*pop + 0.20*proximity) with a per-category diversity cap, then we hand the top 2x to llm.curate_places - one Bedrock call returns the final ordered picks plus a one-line reason each. Names are validated; if the LLM hallucinates we drop those picks and top up from the rule-based shortlist.
    6. Stage 6: distance.haversine_km + budget.estimate_budget annotate every pick. Only places that didn't get a curate-time reason go through llm.generate_place_reasoning.
    7. Stage 7: writes a row to query_history and returns the TripResponse (with meta.intent, meta.llm_curate_used, and meta.degraded so the UI can show whether AI shaped the picks and exactly what degraded).
  5. app/services/intent_parser.py - two-tier intent extractor: rule-based fast path (parse_intent_rule_based) for obvious prompts, LLM escalation (BedrockLLMProvider.extract_intent) for ambiguous ones.
  6. app/clients/places_client.py - runtime circuit breaker (_billing_disabled class flag, is_disabled() classmethod, trip_billing_breaker() trigger).
  7. app/clients/overpass_client.py - mirror failover (OVERPASS_MIRRORS list, _fetch_overpass tries each in order; ping_overpass for the dashboard's health probe).
  8. app/clients/http_base.py: request_json() - shared retries + typed errors. RateLimitError is NOT in the retry tuple - we don't burn quota.
  9. app/db/cache.py - get_fresh_* for TTL-respecting hits, get_stale_* for last-resort fallback.

6. Resilience matrix

Anything outside this process can fail; we never let one upstream take the whole response down.

FailureWhat we doVisible to caller as
Foursquare 429 "no credits" Trip the runtime circuit breaker - subsequent requests skip FSQ entirely - then switch to Overpass for this request meta.degraded = ["places_primary", "places_fallback_overpass"]; subsequent requests carry "places_primary_skipped"
Foursquare 5xx / network Switch to OpenStreetMap Overpass meta.degraded = ["places_primary", "places_fallback_overpass"]
FOURSQUARE_ENABLED=false in .env Skip FSQ entirely; Overpass is the primary meta.degraded += "places_primary_skipped"
Overpass mirror #1 down Try mirror #2 (kumi.systems), then #3 (private.coffee) (transparent - logged at WARNING but no caller-visible tag)
OpenWeather 5xx Serve last cached payload (any age) meta.cache_hits += "weather"
Both places providers down Empty suggestions[], weather + UI still render meta.degraded = ["places_primary", "places", "no_places"]
Nominatim fails (city) Fall back to name-based queries (less precise) meta.degraded += "geocode_city"
Nominatim fails (locality) Use city anchor; no per-place distance / travel cost meta.degraded += "geocode_locality"
LLM curate fails / hallucinates names Use rule-based top-N + per-place reasoning meta.degraded += "llm_curate", llm_curate_used=false
Bedrock per-place reasoning fails Use deterministic rule-based one-liner meta.degraded += "llm"
DB write fails Log + rollback, never block the user response (no tag - we just log)

7. Data model

Three tables, all using SQLAlchemy 2 declarative mapping.

    PLACE_CACHE {
      int id PK
      string city UK "fsqv2:lat,lng:radius:ns | osmv2:lat,lng:radius"
      json payload
      datetime fetched_at
      datetime expires_at "24h TTL"
    }
    WEATHER_LOG {
      int id PK
      string city
      json payload
      datetime fetched_at
      datetime expires_at "30min TTL"
    }

Why one cache table, multiple namespaces? Cache keys carry provider + anchor + intent:

  • fsqv2:17.44,78.38:25000:food - Foursquare, anchor (lat, lng) rounded to 0.01°, radius, intent category. Different intents (food vs history) on the same anchor cache separately.
  • osmv2:17.44,78.38:25000 - Overpass, anchor + radius. Overpass pulls a wide tag set so we don't need to namespace by intent.
  • fsq-name:<city>:<ns> / osm-name:<city> - name-based fallbacks used when geocoding fails.

A stale Foursquare cache will never accidentally satisfy an Overpass request, and a "food" cache will never serve a "history" request, even on the same anchor.

8. Observability

  • Structured JSON logs (one object per line) with ts / level / logger / msg / rid. Easy to grep, easy to ship to CloudWatch.
  • Per-request UUID (rid) injected via a contextvars-based middleware. The same id is sent back as the x-request-id response header so you can line up the browser DevTools network panel with server logs.
  • Pipeline-stage logs - trip_service emits [1/7]... [7/7] prefixed lines so you can trace one request through the orchestrator visually.
  • Two health endpoints - GET /health (cheap, no I/O) and GET /health/detailed (probes every dependency in parallel). Kubernetes liveness vs readiness pattern.
  • In-memory log buffer - GET /logs?limit=200 returns the last 500 lines. The dashboard renders these live - useful in interview demos because failures show up in the panel without leaving the UI.

9. How would you scale this?

  • Move cache to Redis - lower latency than JSONB hits, native TTL, easier per-key invalidation.
  • Bedrock behind a queue - if reasoning grows longer (e.g. multi-place day plans), push to SQS, return immediately, stream partials to the UI via SSE.
  • Per-IP rate limiting - a Redis token bucket in front of /suggest-trip protects upstream quotas (esp. Foursquare).
  • Pre-warm popular cities - a scheduled job calls the pipeline for top-50 cities every night so 100% of user traffic is cache hits.
  • Auth - add a FastAPI dependency that validates a JWT or API key on /suggest-trip; keep /health open for load balancers.

10. Interview cheatsheet

When asked "tell me about this project" - this is the two-minute version:

"It's a FastAPI service that recommends tourist places for a city, tailored to live weather, the user's free-text preference, and the user's locality. The first thing the orchestrator does is translate the prompt - which can be ambiguous like "want to eat something" or "feeling tired" - into a structured intent via a rule-based fast path with a Bedrock fallback for ambiguous phrasings. That intent drives the Foursquare query and the cache namespace, so different prompts genuinely fetch different candidate pools.

In parallel with intent extraction we geocode the city and locality via Nominatim - I treat that as the anchor for everything downstream so we never query providers by a fragile city name. We then fan out to OpenWeather and Foursquare (anchored on ll=lat,lng + 25 km radius) in parallel. Foursquare's free credits run out, so the client carries a class-level circuit-breaker: one "no credits" 429 trips it and every subsequent request in the process skips Foursquare and goes straight to OpenStreetMap Overpass. There's also a hard FOURSQUARE_ENABLED=false kill-switch in the config for the same purpose. Overpass itself fails over across three public mirrors, since the main host is regularly overloaded.

The result pool goes through a hard 30 km filter (so we never surface Lonavala for a Pune query), then a rule-based scorer that combines weather-fit, preference match, popularity, and proximity, plus a diversity cap so we don't hand back five museums in a row. We hand the top 2x to AWS Bedrock (Nova Lite via the Converse API) for the final curate step - one LLM call picks the best matches and writes a one-sentence reason for each. We validate the names against the shortlist so the LLM cannot hallucinate. If the LLM is down we fall through to rule-based ordering with a deterministic fallback sentence. Every degraded path is tagged in meta.degraded, meta.intent shows how the prompt was parsed, and meta.llm_curate_used tells the caller whether AI shaped the picks.

There's a single-page UI on top - vanilla JS + Leaflet - that draws the suggestions on a map with distance lines from the user's locality, plus a system-health panel that probes every dependency in parallel (with three states: ok / disabled / down) and a live in-memory log feed."

Likely follow-ups

  • "Why intent extraction? Couldn't the LLM just rank everything?" Three reasons. (1) Cost / latency - letting the LLM rank means feeding it 80 candidates. Two-step (rules → LLM curate from a 10-item shortlist) is one paid call per request. (2) Determinism - rules cover obvious prompts (food, history) without random LLM variance. (3) Candidate quality - without intent extraction, the LLM is only re-ordering the same generic candidate pool. The original bug was at the fetch layer, not the rank layer. Once Foursquare's query is intent-driven, the candidates themselves match the prompt.
  • "How do you handle 429?" No retry on 429 (quota errors don't recover in 800 ms). The PlacesClient inspects the body - if it contains "no credits" we trip the circuit breaker for the rest of the process and skip Foursquare entirely. Otherwise we switch to the Overpass fallback for this one request.
  • "Why two health endpoints?" Liveness vs readiness - the Kubernetes-standard split. Liveness has no I/O so a load balancer can poll every second; readiness probes every upstream in parallel and is more expensive.
  • "How does the LLM curate avoid hallucinating?" The prompt asks for strict JSON {"picks":[{"name":"...","reason":"..."}]}. Every returned name is validated against the input shortlist - hallucinations are dropped, then we top up from the rule-based shortlist if we end up short.
  • "How does caching work?" Two tables, keyed by provider + anchor + intent (e.g. fsqv2:17.44,78.38:25000:food). Always check fresh first; on upstream failure try stale as a last resort.
  • "Why three Overpass mirrors?" The main overpass-api.de instance is heavily overloaded during peak hours - a single user query can hang for 30s. We try each mirror in turn on transport errors / 5xx / 429. Failover is logged at WARNING but invisible to the caller.
  • "Why diversity in the ranker?" With 80 OSM candidates, the top-K by raw score tends to cluster on the same category (e.g. all monuments). Capping per-category count guarantees the user sees variety - which the interviewer will notice immediately when toggling the preference.
  • "What does meta.intent.source tell me?" "rule" means the keyword fast path matched; "llm" means the rules didn't trigger and Bedrock did the semantic translation; "default" means the prompt was empty. Surfaced in the dashboard's "Understood as" chip so the demo viewer can see which path actually ran.
